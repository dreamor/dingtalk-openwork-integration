# 变更：实现钉钉与大模型交互式对话能力

## Why

当前系统仅支持单轮命令执行模式，用户无法与大模型进行自然的多轮交互会话。这种限制严重降低了用户体验，无法发挥大模型在复杂任务中的优势。参考 OpenClaw 的架构设计，需要实现一个完整的交互式对话系统，支持：

- 多轮对话上下文管理，让用户可以基于之前的对话内容继续交流
- 实时流式响应，让用户可以看到 AI 思考过程的渐进式输出
- 独立的用户会话隔离，确保多用户场景下的数据安全和上下文独立
- 消息去重和流量控制，防止重复消息和并发请求导致的系统过载
- 丰富的文本格式支持，包括 Markdown、代码块等，提升信息呈现质量

## What Changes

本提案实现一个完整的交互式对话系统，参考 OpenClaw 架构：

### 架构设计

```
钉钉用户 ←→ DingTalk Gateway ←→ Stream/WebSocket ←→ Session Manager ←→ Message Queue
                                                                          ↓
                                                               Open Code CLI (Stream)
                                                                          ↓
                                                               流式响应处理 ←→ 响应渲染器
                                                                          ↓
                                                               钉钉用户 (Markdown/代码块)
```

### 核心组件变更

#### 1. 新增 Session Manager (`session-manager/`)

为每个钉钉用户维护独立的对话会话：

- **会话生命周期管理**：创建、维护、过期清理
- **上下文存储**：保存历史消息，支持对话历史检索
- **用户隔离**：不同用户会话数据完全隔离
- **会话状态追踪**：活跃/空闲/过期状态管理

#### 2. 新增 Stream Client (`stream-client/`)

实现与 Open Code CLI 的流式连接：

- **WebSocket/Stream 连接管理**：维持持久连接，处理断线重连
- **实时数据流处理**：边接收边输出，降低首字符延迟
- **流式响应缓冲**：累积响应内容，支持增量渲染
- **连接池管理**：控制并发连接数，优化资源使用

#### 3. 新增 Message Queue (`message-queue/`)

消息去重和并发控制：

- **消息去重**：基于消息 ID 和内容的时间窗口去重
- **流量整形**：令牌桶算法控制请求速率
- **并发限制**：用户级和全局并发限制
- **消息优先级**：区分普通消息和高优先级系统消息

#### 4. 修改 DingTalk Gateway (`dingtalk-gateway/`)

支持交互式对话的消息路由：

- **会话感知路由**：根据会话状态路由消息
- **流式响应推送**：支持 SSE/WebSocket 推送
- **富文本渲染**：Markdown、代码块、表格等格式
- **对话状态管理**：维护每个对话的当前状态

## Impact

### 新增规范

- `session-manager`: 独立用户会话管理
- `stream-client`: Open Code CLI 流式连接客户端
- `message-queue`: 消息队列和并发控制

### 修改规范

- `dingtalk-gateway`: 支持交互式对话流式响应
- `command-executor`: 整合流式执行模式

### 技术依赖

- Open Code CLI 支持 Stream/WebSocket 模式
- 钉钉消息 API 支持 Markdown 格式
- WebSocket 连接支持（可选 SSE 回退）

## 系统行为变更

### 对话模式对比

| 特性 | 当前模式 (单轮) | 新模式 (交互式) |
|------|-----------------|-----------------|
| 上下文 | 无 | 完整对话历史 |
| 响应方式 | 等待完成返回 | 实时流式输出 |
| 消息去重 | 无 | 智能去重 |
| 并发控制 | 无限制 | 用户级限制 |
| 格式支持 | 纯文本 | Markdown/代码块 |

### 用户体验提升

- **自然对话**：用户可以用自然语言持续交流，无需重复描述上下文
- **即时反馈**：看到 AI 思考过程，减少等待焦虑
- **结构化输出**：代码、技术文档等得到正确渲染
- **稳定可靠**：消息不丢失、不重复，系统稳定运行

## 迁移路径

1. **第一阶段**：实现 Session Manager 和基础消息处理
2. **第二阶段**：实现 Stream Client 和流式响应
3. **第三阶段**：实现 Message Queue 和并发控制
4. **第四阶段**：集成 Gateway 和富文本渲染

每个阶段可以独立部署和测试，降低风险。
