## MODIFIED Requirements

### Requirement: 钉钉消息接收与处理

网关 必须 支持交互式对话模式的消息接收和处理。

系统 必须 接收并处理用户通过钉钉发送的消息，将其转换为系统可处理的格式。网关需要智能识别用户意图，将简单的对话消息与命令执行消息区分开来，并进行相应的路由处理。对于交互式对话，网关需要先与 Session Manager 交互获取上下文，再将消息转发至 Message Queue 等待处理。处理完成后，网关还需要将 AI 的响应实时推送回钉钉客户端。

#### Scenario: 接收用户消息

- **WHEN** 用户通过钉钉发送消息
- **THEN** 网关 必须 验证消息签名确保来源合法
- **AND** 网关 必须 解析消息内容提取关键信息
- **AND** 网关 必须 获取发送者的用户标识
- **AND** 网关 必须 添加消息元数据（时间戳、消息 ID）

#### Scenario: 识别会话意图

- **WHEN** 网关接收到用户消息
- **THEN** 系统 必须 根据消息内容判断意图
- **AND** 如果消息为对话意图， 必须 路由到交互式对话流程
- **AND** 如果消息以 `opencode` 开头， 必须 路由到命令执行流程
- **AND** 系统 应该 优先使用交互式对话而非命令执行

#### Scenario: 获取会话上下文

- **WHEN** 需要处理交互式对话消息
- **THEN** 网关 必须 调用 Session Manager 获取会话
- **AND** 如果会话不存在， 必须 创建新会话
- **AND** 网关 必须 将历史消息整合到请求中
- **AND** 网关 必须 将消息添加到会话历史

### Requirement: 流式响应推送

网关 必须 支持通过 SSE 或 WebSocket 向钉钉客户端推送流式响应。

系统 需要实时将 AI 生成的内容推送给用户，提供即时反馈。对于钉钉平台，优先使用 Server-Sent Events (SSE) 实现流式推送，在不支持 SSE 的情况下回退到轮询机制。推送内容需要经过格式化处理，确保 Markdown 和代码块能够正确渲染。网关还需要处理推送过程中的各种异常情况，保证用户体验。

#### Scenario: 建立推送通道

- **WHEN** 用户发起对话请求
- **THEN** 网关 必须 为用户建立推送通道
- **AND** 网关 必须 优先使用 SSE 推送方式
- **AND** 网关 必须 返回推送通道标识给客户端
- **AND** 网关 必须 处理通道建立失败的情况

#### Scenario: 实时内容推送

- **WHEN** AI 生成新的响应内容
- **THEN** 网关 必须 通过推送通道发送内容
- **AND** 网关 应该 在 100ms 内完成推送
- **AND** 系统 必须 支持增量更新推送减少带宽
- **AND** 网关 必须 处理推送延迟和阻塞情况

#### Scenario: 流结束通知

- **WHEN** AI 响应生成完成
- **THEN** 网关 必须 发送流结束标记
- **AND** 网关 必须 关闭推送通道
- **AND** 网关 必须 清理相关资源

### Requirement: 富文本格式支持

网关 必须 支持 Markdown 和代码块等富文本格式的正确渲染。

AI 响应通常包含 Markdown 格式的内容，包括标题、列表、代码块、表格等。网关需要将原始响应转换为钉钉支持的消息格式，确保这些内容能够正确显示。特别是代码块需要语法高亮，表格需要格式化，行代码需要特别标记。网关还需要处理内容溢出和格式嵌套等复杂情况。

#### Scenario: Markdown 解析

- **WHEN** 收到包含 Markdown 的响应
- **THEN** 网关 必须 正确解析 Markdown 结构
- **AND** 标题 必须 转换为适当的格式
- **AND** 列表 必须 按层级渲染
- **AND** 引用块 必须 正确缩进显示

#### Scenario: 代码块渲染

- **WHEN** 响应包含代码块
- **THEN** 网关 必须 识别代码块的起止位置
- **AND** 网关 应该 检测编程语言
- **AND** 网关 必须 使用代码块格式发送
- **AND** 网关 应该 添加行号便于阅读

#### Scenario: 行内代码处理

- **WHEN** 响应包含行内代码片段
- **THEN** 网关 必须 识别行内代码标记
- **AND** 网关 必须 使用等宽字体格式发送
- **AND** 网关 必须 正确处理转义字符

#### Scenario: 表格格式化

- **WHEN** 响应包含 Markdown 表格
- **THEN** 网关 必须 解析表格结构
- **AND** 网关 必须 使用等宽字体对齐列
- **AND** 表格 必须 保持行列对齐

### Requirement: 对话状态管理

网关 必须 维护每个对话的当前状态，支持状态转换和恢复。

交互式对话涉及多个状态转换，网关需要准确追踪每个对话的状态，包括初始化、等待响应、响应中、已完成等状态。状态信息需要在用户重新连接时能够恢复，确保对话连续性。网关还需要处理状态异常和跳转情况。

#### Scenario: 状态转换

- **WHEN** 对话状态发生变化
- **THEN** 网关 必须 记录当前状态
- **AND** 网关 必须 触发相应的状态处理逻辑
- **AND** 网关 必须 通知相关组件状态变化
- **AND** 系统 必须 记录完整的状态转换历史

#### Scenario: 连接恢复

- **WHEN** 用户重新连接（断线后重连）
- **THEN** 网关 必须 恢复用户对话状态
- **AND** 网关 必须 重建推送通道
- **AND** 网关 必须 补发未完成响应的内容
- **AND** 如果响应已完成， 必须 返回最终结果

#### Scenario: 状态持久化

- **WHEN** 对话达到检查点
- **THEN** 网关 必须 持久化当前状态
- **AND** 系统 必须 支持从持久化状态恢复
- **AND** 状态持久化 必须 不影响响应速度
