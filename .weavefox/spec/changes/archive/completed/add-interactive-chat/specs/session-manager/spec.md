## ADDED Requirements

### Requirement: 会话管理

系统 必须 为每个钉钉用户维护独立的对话会话，以支持多轮对话和上下文关联。

#### Scenario: 创建新会话

- **WHEN** 用户发送第一条消息且没有现有会话
- **THEN** 系统 必须 为用户创建一个新的会话
- **AND** 系统 必须 生成唯一的 `conversationId`
- **AND** 系统 必须 初始化空的消息历史列表
- **AND** 系统 必须 设置会话状态为 `Active`

#### Scenario: 恢复已有会话

- **WHEN** 用户在有效会话时间内发送消息
- **THEN** 系统 必须 根据用户 ID 获取已有会话
- **AND** 系统 必须 将消息追加到会话的消息历史
- **AND** 系统 必须 将会话状态更新为 `Active`

#### Scenario: 会话自动过期

- **WHEN** 会话超过配置的 TTL (默认 30 分钟) 无活动
- **THEN** 系统 必须 自动将会话状态标记为 `Expired`
- **AND** 系统 必须 清理过期的会话数据
- **AND** 后续用户消息 必须 创建新的会话

#### Scenario: 会手动结束会话

- **WHEN** 用户发送结束会话命令
- **THEN** 系统 必须 将会话状态标记为 `Terminated`
- **AND** 系统 必须 保留会话历史用于审计
- **AND** 后续用户消息 必须 创建新的会话

### Requirement: 消息历史管理

系统 必须 维护完整的消息历史，支持检索和上下文重建。

#### Scenario: 消息追加

- **WHEN** 用户发送消息或收到 AI 响应
- **THEN** 系统 必须 将消息追加到会话历史
- **AND** 系统 必须 记录消息的完整元数据
  - `messageId`: 唯一标识符
  - `timestamp`: 发送时间
  - `role`: 发送者角色 (user/assistant/system)
  - `content`: 消息内容
  - `metadata`: 可选的附加信息

#### Scenario: 获取对话历史

- **WHEN** 需要获取会话的消息历史
- **THEN** 系统 必须 按时间顺序返回消息列表
- **AND** 系统 必须 支持限制返回的消息数量
- **AND** 系统 必须 支持按关键词搜索历史消息

#### Scenario: 历史消息裁剪

- **WHEN** 消息历史超过最大 token 限制
- **THEN** 系统 必须 保留最近和最相关的消息
- **AND** 系统 应该 使用摘要算法压缩早期消息
- **AND** 系统 必须 确保不丢失关键上下文信息

### Requirement: 用户会话隔离

系统 必须 确保不同用户的会话数据完全隔离。

#### Scenario: 独立数据存储

- **WHEN** 用户 A 和用户 B 同时使用系统
- **THEN** 系统 必须 为每个用户维护独立的会话数据
- **AND** 系统 必须 防止用户访问他人的会话历史
- **AND** 系统 必须 在用户级别强制数据隔离

#### Scenario: 并发会话安全

- **WHEN** 同一用户的多个客户端同时发送消息
- **THEN** 系统 必须 处理并发请求而不导致数据损坏
- **AND** 系统 必须 使用适当的锁机制保护会话数据
- **AND** 系统 必须 按消息到达顺序处理请求

### Requirement: 会话状态追踪

系统 必须 准确追踪每个会话的当前状态。

#### Scenario: 状态转换

- **WHEN** 会话状态发生变化
- **THEN** 系统 必须 记录状态转换的完整历史
- **AND** 系统 必须 支持的状态转换路径：
  - `Creating` → `Active`
  - `Active` → `Idle` (超时后)
  - `Idle` → `Active` (收到消息)
  - `Active` → `Expired` (TTL 过期)
  - `Active` → `Terminated` (手动结束)

#### Scenario: 状态持久化

- **WHEN** 会话状态需要恢复或迁移
- **THEN** 系统 必须 能够将状态持久化到存储介质
- **AND** 系统 必须 支持状态恢复时的数据一致性检查
- **AND** 系统 必须 提供状态数据的备份和恢复机制
