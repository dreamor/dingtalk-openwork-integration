## ADDED Requirements

### Requirement: 消息去重

系统 必须 去除重复消息，防止重复处理导致数据不一致。

在实际使用中，由于网络重试或用户误操作，同一消息可能被发送多次。系统需要在入口处进行去重处理，避免重复处理。去重策略需要考虑消息内容相似度、时间窗口和用户发送频率等因素，在保证不丢失重要消息的同时有效过滤重复消息。

#### Scenario: 基于消息 ID 去重

- **WHEN** 接收到具有相同消息 ID 的消息
- **THEN** 系统 必须 识别为重复消息
- **AND** 系统 必须 丢弃该消息
- **AND** 系统 必须 返回已处理的响应缓存

#### Scenario: 基于内容去重

- **WHEN** 接收到内容相似且时间相近的消息
- **THEN** 系统 必须 计算消息内容相似度
- **AND** 如果相似度超过阈值（90%）， 必须 视为重复
- **AND** 系统 应该 结合用户ID进行去重

#### Scenario: 时间窗口去重

- **WHEN** 在配置的窗口期内收到重复消息
- **THEN** 系统 必须 在时间窗口内去重有效
- **AND** 系统 必须 支持配置时间窗口大小（默认 5 秒）
- **AND** 超出时间窗口的消息 必须 作为新消息处理

### Requirement: 流量控制

系统 必须 实现流量控制，防止系统过载和滥用。

用户可能频繁发送消息或短时间内发送大量请求，系统需要实现流量控制机制来保护系统稳定性。流量控制采用令牌桶算法，支持用户级和全局两个维度的限制。当用户超出配额时，系统需要返回明确的限流响应，让用户了解需要等待的时间。

#### Scenario: 令牌桶限流

- **WHEN** 用户发送消息
- **THEN** 系统 必须 检查令牌桶剩余令牌
- **AND** 如果有可用令牌， 必须 消耗一个令牌并处理消息
- **AND** 如果没有可用令牌， 必须 拒绝消息并返回限流响应
- **AND** 系统 必须 以固定速率补充令牌

#### Scenario: 用户级限流

- **WHEN** 单个用户发送消息频率过高
- **THEN** 系统 必须 检查用户级配额
- **AND** 系统 必须 限制用户每分钟请求数（默认 60/分钟）
- **AND** 超出配额的消息 必须 放入等待队列或拒绝

#### Scenario: 全局限流

- **WHEN** 系统整体请求量达到上限
- **THEN** 系统 必须 启用全局限流
- **AND** 系统 必须 限制全局并发请求数（默认 100）
- **AND** 系统 应该 优先处理高优先级用户的消息

#### Scenario: 限流响应

- **WHEN** 消息被限流拒绝
- **THEN** 系统 必须 返回明确的限流错误
- **AND** 响应 必须 包含建议的重试时间
- **AND** 系统 应该 提供队列位置信息

### Requirement: 并发控制

系统 必须 控制并发请求数量，确保系统稳定运行。

高并发场景下，大量同时到达的请求可能导致系统资源耗尽。系统需要实现并发控制机制，限制同时处理的请求数量，并使用队列管理超出并发限制的请求。并发控制需要在用户隔离和系统效率之间取得平衡。

#### Scenario: 用户级并发限制

- **WHEN** 单个用户同时有多个活跃请求
- **THEN** 系统 必须 限制用户并发数（默认 3）
- **AND** 超出限制的请求 必须 进入等待队列
- **AND** 系统 必须 按 FIFO 顺序处理排队请求

#### Scenario: 全局并发限制

- **WHEN** 系统同时处理的请求数达到上限
- **THEN** 系统 必须 停止接受新请求
- **AND** 新请求 必须 进入全局等待队列
- **AND** 系统 必须 监控并发状态并报告

#### Scenario: 请求排队

- **WHEN** 请求因并发限制需要等待
- **THEN** 系统 必须 通知用户请求已排队
- **AND** 系统 必须 提供预估等待时间
- **AND** 系统 必须 设置最大排队时间（默认 60 秒）

#### Scenario: 超时处理

- **WHEN** 排队请求超过最大等待时间
- **THEN** 系统 必须 返回超时错误
- **AND** 系统 必须 从队列中移除该请求
- **AND** 系统 应该 建议用户稍后重试

### Requirement: 消息优先级

系统 必须 支持消息优先级，确保重要消息优先处理。

不同类型的消息有不同的紧急程度，用户消息应该优先处理，而系统维护任务可以在闲时处理。系统需要实现优先级队列，支持多级优先级，并在高优先级消息到达时进行抢占。

#### Scenario: 优先级队列

- **WHEN** 消息进入队列
- **THEN** 系统 必须 根据优先级排序
- **AND** 不同优先级的消息 必须 按正确顺序处理
- **AND** 相同优先级的消息 必须 按到达顺序处理

#### Scenario: 优先级分配

- **WHEN** 消息到达
- **THEN** 系统 必须 根据消息类型分配优先级
- **AND** 用户直接消息 应 为 `High` 优先级
- **AND** 重试消息 应 为 `Medium` 优先级
- **AND** 系统维护消息 应 为 `Low` 优先级

#### Scenario: 优先级抢占

- **WHEN** 高优先级消息到达
- **THEN** 系统 必须 暂停当前低优先级处理（可选）
- **AND** 系统 必须 优先处理高优先级消息
- **AND** 系统 应该 记录抢占事件用于审计

#### Scenario: 优先级降级

- **WHEN** 消息在队列中等待过长
- **THEN** 系统 应该 提升消息优先级
- **AND** 系统 应该 防止优先级反转
- **AND** 长期等待消息 必须 得到处理机会
