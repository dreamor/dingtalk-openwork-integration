# 变更：将钉钉消息接收从 Webhook 回调模式改为轮询拉取模式

## Why

当前系统使用 Webhook 回调模式接收钉钉消息，需要通过内网穿透工具（如 ngrok、frp）将本地服务暴露到公网，才能接收钉钉服务器推送的消息。这种方式存在以下问题：

1. **部署复杂**：需要在公网部署内网穿透服务或使用云服务器
2. **运维成本**：需要维护内网穿透服务的稳定性和安全性
3. **连接不稳定**：内网穿透链路可能不稳定，导致消息丢失
4. **安全风险**：公网暴露增加了安全攻击面

改为轮询模式后，本地服务可以主动从钉钉拉取消息，无需公网暴露，更加安全和稳定。

## What Changes

- **移除 Webhook 依赖**：不再需要暴露 `/api/dingtalk/webhook` 端点给公网
- **增加轮询服务**：实现定时任务主动拉取钉钉消息
- **实现消息游标**：记录上次拉取位置，避免重复处理消息
- **优化资源管理**：轮询间隔动态调整，根据消息频率自适应

### 核心变更点

1. 新增 `PollingService` 服务，实现消息主动拉取
2. 新增 `MessageCursor` 管理，记录消息拉取位置
3. 修改 `DingtalkService` 添加消息拉取相关 API
4. 改造 `GatewayServer`，可选保留 Webhook 作为回退
5. 新增轮询配置项：间隔时间、超时设置、失败重试

## Impact

### 受影响规范

- `specs/dingtalk/`: 钉钉集成模块
- `specs/gateway/`: 网关服务模块

### 受影响代码

- `src/dingtalk/dingtalk.ts`: 新增消息拉取 API
- `src/gateway/index.ts`: 移除/保留 Webhook 端点
- `src/config.ts`: 新增轮询配置项
- `src/index.ts`: 启动轮询服务

### 配置变更

需要新增以下环境变量：
- `POLLING_INTERVAL`: 轮询间隔（默认 3000ms）
- `POLLING_TIMEOUT`: 单次拉取超时（默认 5000ms）
- `POLLING_ENABLED`: 是否启用轮询模式（默认为 true）

### 兼容性

- 新模式默认启用，Webhook 端点保留但标记为废弃
- 支持配置切换回传统模式（兼容旧部署）
- 消息处理逻辑保持不变
