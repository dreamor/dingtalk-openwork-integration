## ADDED Requirements

### Requirement: 钉钉消息轮询拉取

系统 必须 实现主动轮询拉取钉钉消息的功能，替代传统 Webhook 回调模式，实现无需内网穿透的消息接收。

#### Scenario: 基础消息拉取

- **WHEN** 轮询服务启动时
- **THEN** 系统 按照配置的间隔时间向钉钉消息拉取 API 发起请求
- **AND** 系统 应该 返回指定时间窗口内的新消息
- **AND** 每次拉取最多返回配置的最大消息数

#### Scenario: 消息游标管理

- **WHEN** 系统成功拉取消息后
- **THEN** 系统 必须 记录最后处理的消息 ID 和时间戳
- **AND** 下次拉取时 从记录的位置继续获取新消息
- **AND** 系统 应该 支持从指定游标位置恢复拉取

#### Scenario: 空响应处理

- **WHEN** 拉取请求返回无新消息
- **THEN** 系统 应该 继续等待下一个轮询周期
- **AND** 系统 可以 根据空闲时间适当延长轮询间隔

#### Scenario: 拉取失败处理

- **WHEN** 消息拉取 API 调用失败
- **THEN** 系统 应该 记录错误日志
- **AND** 系统 应该 在配置的退避时间后重试
- **AND** 连续失败超过阈值后 应该 增加重试间隔

---

### Requirement: 动态轮询间隔

系统 应该 实现动态调整轮询间隔的功能，在消息实时性和资源消耗之间取得平衡。

#### Scenario: 有消息时缩短间隔

- **WHEN** 拉取到新消息时
- **THEN** 系统 应该 将轮询间隔调整为配置的最小间隔
- **AND** 系统 应该 优先保证消息的实时性

#### Scenario: 空闲时延长间隔

- **WHEN** 连续 N 次拉取无新消息
- **THEN** 系统 应该 逐步延长轮询间隔
- **AND** 间隔延长不超过配置的最大间隔
- **AND** 系统 应该 在指定空闲超时后进入低频模式

#### Scenario: 恢复有消息时

- **WHEN** 低频模式下拉取到新消息
- **THEN** 系统 应该 立即恢复为最小轮询间隔
- **AND** 系统 应该 确保消息处理的实时性

---

### Requirement: 轮询配置管理

系统 必须 提供轮询相关的配置项，支持灵活调整轮询行为。

#### Scenario: 环境变量配置

- **WHEN** 应用程序启动时
- **THEN** 系统 必须 从环境变量读取轮询配置
- **AND** 未配置时 使用默认值
- **AND** 支持的配置项包括：
  - POLLING_INTERVAL（基础间隔）
  - POLLING_TIMEOUT（API 超时）
  - POLLING_ENABLED（是否启用）

#### Scenario: 配置验证

- **WHEN** 轮询配置加载完成后
- **THEN** 系统 必须 验证配置的有效性
- **AND** 最小间隔必须小于最大间隔
- **AND** 超时时间必须大于 0
- **AND** 验证失败时 使用默认值或记录警告

## MODIFIED Requirements

### Requirement: 钉钉消息接收

系统 必须 支持通过轮询模式接收钉钉消息，Webhook 端点标记为可选。

#### Scenario: 轮询模式默认启用

- **WHEN** 应用程序启动时
- **THEN** 系统 默认启用轮询模式主动拉取消息
- **AND** 系统 不需要 公网暴露即可接收消息
- **AND** 不再需要配置内网穿透服务

#### Scenario: Webhook 端点兼容（可选）

- **WHEN** 配置项 WEBHOOK_ENABLED 为 true 时
- **THEN** 系统 应该 保留 Webhook 端点
- **AND** 系统 应该 记录 Webhook 模式已废弃的警告
- **AND** 用户可以切换回传统 Webhook 模式

#### Scenario: 消息处理一致性

- **WHEN** 通过轮询模式接收消息
- **THEN** 系统 必须 使用与原 Webhook 模式相同的消息处理流程
- **AND** 消息去重逻辑保持一致
- **AND** 流量控制逻辑保持一致
- **AND** 会话管理逻辑保持一致
